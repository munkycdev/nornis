# Requirements Document

## Introduction

This feature implements the Review Proposal Workflow — the API and application service layer for reviewing, accepting, rejecting, and editing ReviewProposals generated by the async source extraction worker. When proposals are accepted, the system creates or updates Artifacts, ArtifactFacts, and ArtifactRelationships in the campaign knowledge graph and creates SourceReferences for traceability. This is the critical bridge between AI-generated proposals and accepted campaign knowledge — the point where "AI proposes, users decide" becomes reality.

The feature covers: listing the review queue filtered by authorization and visibility; accepting proposals (applying the proposed mutation to the knowledge graph); rejecting proposals; editing proposals before acceptance; batch operations for efficiency; ReviewBatch status lifecycle management; and idempotency for safe retries.

## Glossary

- **Review_Service**: The application-layer service responsible for orchestrating proposal review operations including authorization, visibility enforcement, proposal application, and status transitions
- **ReviewBatch**: A record grouping all proposals from a single extraction run for one source, with status lifecycle Pending → InReview → Completed
- **ReviewProposal**: An individual reviewable change proposal extracted from a source — one proposal per discrete knowledge graph mutation
- **ProposedValueJson**: The JSON payload within a ReviewProposal describing the proposed artifact, fact, or relationship mutation in a schema-specific format
- **ChangeType**: One of CreateArtifact, UpdateArtifact, MergeArtifact, AddFact, UpdateFact, AddRelationship, or UpdateRelationship — classifies the mutation a proposal would apply
- **TargetType**: One of Artifact, ArtifactFact, or ArtifactRelationship — identifies the entity type being mutated
- **Proposal_Status**: One of Pending, Accepted, Rejected, or Edited — the review state of an individual proposal
- **Batch_Status**: One of Pending, InReview, Completed, Canceled, or Failed — the lifecycle state of a ReviewBatch
- **GM**: Game Master role — can review all proposals in the campaign regardless of source author
- **Player**: Player role — can review proposals generated from sources the Player created
- **Observer**: Observer role — cannot review proposals
- **VisibilityScope**: One of Private, GMOnly, or PartyVisible — determines which campaign members can see proposals and their resulting knowledge
- **SourceReference**: A citation linking an accepted artifact, fact, or relationship back to the source that produced it
- **Artifact**: A structured campaign knowledge entity (Character, Location, Item, Faction, Event, Thread, Concept, Document)
- **ArtifactFact**: An atomic statement about an artifact with predicate, value, truth state, and visibility
- **ArtifactRelationship**: A bidirectional typed edge between two artifacts with description, confidence, truth state, and visibility
- **Knowledge_Graph**: The collection of Artifacts, ArtifactFacts, and ArtifactRelationships representing accepted campaign knowledge
- **Campaign_Member**: A user with an active membership in the target campaign, resolved via CampaignMemberActionFilter

## Requirements

### Requirement 1: List Review Queue

**User Story:** As a GM or Player, I want to see my pending review proposals filtered by my campaign role and visibility permissions, so that I can review AI-generated proposals relevant to my access level.

#### Acceptance Criteria

1. WHEN a GM requests the review queue for a campaign, THE Review_Service SHALL return all ReviewProposals in Pending status across all ReviewBatches in the campaign, regardless of which user created the source
2. WHEN a Player requests the review queue for a campaign, THE Review_Service SHALL return only ReviewProposals in Pending status from ReviewBatches associated with sources the Player created (CreatedByUserId matches the requesting user)
3. WHEN an Observer requests the review queue for a campaign, THE Review_Service SHALL return an empty list
4. THE Review_Service SHALL NOT return proposals derived from GMOnly sources to Players or Observers
5. THE Review_Service SHALL NOT return proposals derived from Private sources to any user other than the source creator and campaign GMs
6. THE Review_Service SHALL return proposals derived from PartyVisible sources to all authorized reviewers (GMs and the source-creating Player)
7. THE Review_Service SHALL return each proposal with its Id, ReviewBatchId, ChangeType, TargetType, TargetId, ProposedValueJson, Rationale, Confidence, Status, and CreatedAt
8. THE Review_Service SHALL return proposals ordered by CreatedAt ascending within each ReviewBatch, with ReviewBatches ordered by CreatedAt ascending
9. WHEN an authorized reviewer requests proposals filtered by a specific ReviewBatchId, THE Review_Service SHALL return only proposals within that batch that the reviewer is authorized to see according to role-based and visibility rules (criteria 1-6)
10. IF the specified ReviewBatchId does not exist within the campaign, THEN THE Review_Service SHALL return a not-found error
11. THE Review_Service SHALL return a maximum of 200 proposals per request; IF the review queue contains more than 200 matching proposals, THEN THE Review_Service SHALL return the first 200 proposals according to the defined ordering and include an indicator that additional proposals exist
12. IF the review queue for the requesting user contains zero matching proposals, THEN THE Review_Service SHALL return an empty list with no error

### Requirement 2: Accept Proposal

**User Story:** As a GM or authorized Player, I want to accept a review proposal, so that the proposed change is applied to the campaign knowledge graph.

#### Acceptance Criteria

1. WHEN an authorized reviewer accepts a proposal with Status Pending or Edited, THE Review_Service SHALL apply the proposed mutation to the Knowledge_Graph, transition the proposal Status to Accepted, set ReviewedAt to the current UTC timestamp, and set ReviewedByUserId to the acting user
2. WHEN the ChangeType is CreateArtifact, THE Review_Service SHALL create a new Artifact with fields from ProposedValueJson including Name, Type, Summary, Visibility, Confidence, and Status set to Active, with CampaignId from the ReviewBatch and CreatedAt and UpdatedAt set to the current UTC timestamp
3. WHEN the ChangeType is UpdateArtifact and TargetId references an existing Artifact in the campaign, THE Review_Service SHALL update the Artifact fields specified in ProposedValueJson (Name, Summary, Visibility, Confidence, Status) and set UpdatedAt to the current UTC timestamp
4. WHEN the ChangeType is AddFact, THE Review_Service SHALL create a new ArtifactFact with fields from ProposedValueJson including ArtifactId (resolved from TargetId or a newly created Artifact), Predicate, Value, Confidence, TruthState, and Visibility, with CreatedAt and UpdatedAt set to the current UTC timestamp
5. WHEN the ChangeType is UpdateFact and TargetId references an existing ArtifactFact, THE Review_Service SHALL update the ArtifactFact fields specified in ProposedValueJson (Value, Confidence, TruthState, Visibility) and set UpdatedAt to the current UTC timestamp
6. WHEN the ChangeType is AddRelationship, THE Review_Service SHALL create a new ArtifactRelationship with fields from ProposedValueJson including CampaignId, ArtifactAId, ArtifactBId, Type, Description, Confidence, TruthState, and Visibility, with CreatedAt and UpdatedAt set to the current UTC timestamp
7. WHEN the ChangeType is UpdateRelationship and TargetId references an existing ArtifactRelationship, THE Review_Service SHALL update the ArtifactRelationship fields specified in ProposedValueJson (Type, Description, Confidence, TruthState, Visibility) and set UpdatedAt to the current UTC timestamp
8. WHEN a proposal is accepted, THE Review_Service SHALL create a SourceReference linking the source (from the ReviewBatch) to the created or updated entity with TargetType set to the appropriate SourceReferenceTargetType (Artifact, ArtifactFact, or ArtifactRelationship) and TargetId set to the entity Id
9. IF the TargetId on an UpdateArtifact, UpdateFact, or UpdateRelationship proposal does not reference an existing entity in the campaign, THEN THE Review_Service SHALL return a validation error indicating the target entity was not found
10. IF the ProposedValueJson cannot be deserialized into the expected schema for the specified ChangeType, THEN THE Review_Service SHALL return a validation error indicating the proposal payload is malformed
11. IF the specified proposal does not exist within the campaign, THEN THE Review_Service SHALL return a not-found error
12. IF the specified proposal Status is Accepted or Rejected, THEN THE Review_Service SHALL return an error indicating that only Pending or Edited proposals can be accepted

### Requirement 3: Reject Proposal

**User Story:** As a GM or authorized Player, I want to reject a review proposal, so that the proposed change is not applied to the campaign knowledge graph.

#### Acceptance Criteria

1. WHEN an authorized reviewer rejects a proposal with Status Pending or Edited, THE Review_Service SHALL transition the proposal Status to Rejected, set ReviewedAt to the current UTC timestamp, and set ReviewedByUserId to the acting user
2. WHEN a proposal is rejected, THE Review_Service SHALL NOT create, update, or delete any Artifact, ArtifactFact, ArtifactRelationship, or SourceReference record
3. IF the specified proposal does not exist within the campaign, THEN THE Review_Service SHALL return a not-found error
4. IF the specified proposal Status is Accepted, THEN THE Review_Service SHALL return an error indicating that Accepted proposals cannot be rejected
5. IF a user attempts to reject a proposal they cannot see due to visibility rules, THEN THE Review_Service SHALL return a not-found error rather than a forbidden error
6. WHEN a proposal is rejected and it is the first proposal reviewed in its ReviewBatch, THE Review_Service SHALL transition the ReviewBatch Status from Pending to InReview

### Requirement 4: Edit Proposal

**User Story:** As a GM or authorized Player, I want to edit a proposal's ProposedValueJson before accepting it, so that I can correct or refine AI-generated values before they become accepted knowledge.

#### Acceptance Criteria

1. WHEN an authorized reviewer submits an edit request for a proposal with Status Pending or Edited, THE Review_Service SHALL validate the new ProposedValueJson against the expected schema for the proposal's ChangeType, replace the entire ProposedValueJson on the proposal with the submitted value, transition the proposal Status to Edited, set ReviewedAt to the current UTC timestamp, and set ReviewedByUserId to the acting user
2. WHEN a proposal has Status Edited, THE Review_Service SHALL allow that proposal to be subsequently accepted (applying the edited ProposedValueJson following the same application logic as Requirement 2) or rejected (following the same logic as Requirement 3)
3. IF the new ProposedValueJson does not conform to the expected schema for the specified ChangeType or exceeds 32,768 characters in length, THEN THE Review_Service SHALL return a validation error indicating which fields are invalid or that the payload exceeds the maximum allowed size
4. IF the specified proposal does not exist within the campaign, THEN THE Review_Service SHALL return a not-found error
5. IF the specified proposal Status is Accepted or Rejected, THEN THE Review_Service SHALL return an error indicating that only Pending or Edited proposals can be edited
6. THE Review_Service SHALL NOT apply the mutation to the Knowledge_Graph when a proposal is edited — application occurs only upon explicit acceptance

### Requirement 5: Batch Operations

**User Story:** As a GM or authorized Player, I want to accept or reject multiple proposals at once, so that I can efficiently process review queues with many proposals.

#### Acceptance Criteria

1. WHEN an authorized reviewer submits a batch accept request with a list of proposal Ids, THE Review_Service SHALL accept each proposal individually following the same logic as single-proposal acceptance (Requirement 2)
2. WHEN an authorized reviewer submits a batch reject request with a list of proposal Ids, THE Review_Service SHALL reject each proposal individually following the same logic as single-proposal rejection (Requirement 3)
3. IF any proposal in a batch operation fails validation (target not found, malformed payload, or unauthorized), THEN THE Review_Service SHALL skip that proposal, continue processing remaining proposals, and return the list of failed proposal Ids with their error reasons alongside the list of successfully processed proposal Ids; proposals that are already in the matching terminal state (Accepted for batch accept, Rejected for batch reject) SHALL be treated as idempotent successes per Requirement 10 and included in the successfully processed list without reapplication; proposals in a non-matching terminal state or in Edited status when not eligible SHALL be reported as failed with the reason
4. IF a batch request contains fewer than 1 or more than 50 proposal Ids, THEN THE Review_Service SHALL return a validation error without processing any proposals; IF a batch request contains duplicate proposal Ids, THEN THE Review_Service SHALL deduplicate the list and process each unique proposal Id exactly once
5. THE Review_Service SHALL verify authorization for each proposal in the batch — a Player SHALL only be able to accept or reject proposals from sources the Player created
6. THE Review_Service SHALL process batch operations sequentially in the order the proposal Ids appear in the request to maintain consistency when proposals reference shared artifacts

### Requirement 6: Authorization

**User Story:** As a system operator, I want proposal review operations to enforce campaign role and source ownership rules, so that only authorized users can accept, reject, or edit proposals.

#### Acceptance Criteria

1. WHEN a GM submits a review operation (accept, reject, or edit) for any proposal in the campaign, THE Review_Service SHALL allow the operation regardless of which user created the source
2. WHEN a Player submits a review operation for a proposal, THE Review_Service SHALL allow the operation only if the source associated with the proposal's ReviewBatch was created by the Player (Source.CreatedByUserId matches the requesting user's Id as derived from JWT claims)
3. IF a Player submits a review operation for a proposal whose ReviewBatch is associated with a source created by a different user, THEN THE Review_Service SHALL return HTTP 403 Forbidden with an error indicating insufficient permissions
4. IF an Observer submits any review operation (accept, reject, or edit), THEN THE Review_Service SHALL return HTTP 403 Forbidden with an error indicating insufficient permissions
5. THE Review_Service SHALL derive the acting user identity from the authenticated JWT claims and SHALL NOT accept client-provided user identifiers for review operations
6. THE API SHALL require campaign membership for all review endpoints, enforced via the CampaignMemberActionFilter, before evaluating role-based authorization
7. IF the source or ReviewBatch associated with a proposal cannot be resolved during authorization evaluation, THEN THE Review_Service SHALL deny the operation and return an error indicating the proposal cannot be authorized

### Requirement 7: Visibility Enforcement

**User Story:** As a system operator, I want visibility rules enforced on the review queue, so that proposals derived from private or GM-only sources are never exposed to unauthorized members.

#### Acceptance Criteria

1. THE Review_Service SHALL filter proposals in all query and operation responses based on the VisibilityScope of the source associated with the proposal's ReviewBatch, such that Private proposals are visible only to the source creator and campaign GMs, GMOnly proposals are visible only to campaign GMs, and PartyVisible proposals are visible to all campaign members with reviewer access (GMs and the source-creating Player)
2. THE Review_Service SHALL NOT return proposals derived from Private sources to any user other than the source creator and campaign GMs
3. THE Review_Service SHALL NOT return proposals derived from GMOnly sources to Players or Observers
4. WHEN a user attempts to accept, reject, or edit a proposal they cannot see due to visibility rules, THE Review_Service SHALL respond as if the proposal does not exist (not-found error rather than forbidden)
5. WHEN a proposal is accepted and creates or updates knowledge graph entities, THE Review_Service SHALL set the Visibility on created entities to match the VisibilityScope specified in ProposedValueJson; IF ProposedValueJson does not specify a VisibilityScope, THEN THE Review_Service SHALL default the entity Visibility to the VisibilityScope of the source associated with the proposal's ReviewBatch
6. WHEN a batch operation includes proposal Ids that the requesting user cannot see due to visibility rules, THE Review_Service SHALL treat each invisible proposal as not found, report it as failed with a not-found reason, and continue processing the remaining visible proposals

### Requirement 8: ReviewBatch Status Transitions

**User Story:** As a system operator, I want ReviewBatch status to reflect the aggregate progress of its proposals, so that users and the system can determine batch completeness.

#### Acceptance Criteria

1. WHEN a proposal in a ReviewBatch with Batch_Status Pending is accepted, rejected, or edited, THE Review_Service SHALL transition the ReviewBatch Status from Pending to InReview
2. WHEN all proposals in a ReviewBatch have reached a terminal Proposal_Status (each has Status Accepted or Rejected), THE Review_Service SHALL transition the ReviewBatch Status from InReview to Completed and set CompletedAt to the current UTC timestamp
3. THE Review_Service SHALL NOT transition a ReviewBatch to Completed while any proposal in the batch remains in Pending or Edited status
4. IF a ReviewBatch already has Status InReview when a proposal is reviewed, THEN THE Review_Service SHALL leave the batch Status as InReview until all proposals have reached a terminal Proposal_Status
5. IF a ReviewBatch has Batch_Status Canceled or Failed, THEN THE Review_Service SHALL NOT transition the batch to InReview or Completed regardless of proposal status changes
6. IF a ReviewBatch contains zero proposals and its Status is Pending, THEN THE Review_Service SHALL remain in Pending status until proposals are added or the batch is explicitly Canceled or Failed

### Requirement 9: Proposal Application Logic

**User Story:** As the system, I want proposal acceptance to correctly mutate the knowledge graph based on ChangeType, so that accepted proposals create accurate and consistent campaign knowledge.

#### Acceptance Criteria

1. WHEN accepting a CreateArtifact proposal, THE Review_Service SHALL create the Artifact, set the ReviewProposal TargetId to the newly created Artifact Id, and use that Id as the TargetId for the corresponding SourceReference
2. WHEN accepting an AddFact proposal where the TargetId references an Artifact that exists in the campaign, THE Review_Service SHALL create the ArtifactFact with ArtifactId set to the TargetId
3. WHEN accepting an AddRelationship proposal where both ArtifactAId and ArtifactBId reference existing Artifacts in the campaign, THE Review_Service SHALL create the ArtifactRelationship with the specified artifact references
4. IF an AddRelationship proposal references an ArtifactAId or ArtifactBId that does not exist in the campaign, THEN THE Review_Service SHALL return a validation error indicating which artifact reference was not found
5. WHEN accepting a MergeArtifact proposal, THE Review_Service SHALL identify the target Artifact from the ReviewProposal TargetId and the source Artifact from the SourceArtifactId field in ProposedValueJson, update the target Artifact fields (Name, Summary, Visibility, Confidence) from ProposedValueJson, reassign all ArtifactFacts and ArtifactRelationships from the source Artifact to the target Artifact, remove any ArtifactRelationships that would become self-referencing after reassignment, and set the source Artifact Status to Archived
6. IF a MergeArtifact proposal references a TargetId or SourceArtifactId that does not exist in the campaign, THEN THE Review_Service SHALL return a validation error indicating which artifact was not found
7. WHEN accepting any proposal, THE Review_Service SHALL create the knowledge graph entity, the SourceReference, and update the proposal status within a single atomic operation — if any step fails, none of the changes from that acceptance SHALL be committed
8. IF an AddFact proposal references a TargetId (ArtifactId) that does not exist in the campaign, THEN THE Review_Service SHALL return a validation error indicating the target artifact was not found

### Requirement 10: Idempotency

**User Story:** As a system operator, I want accepting an already-accepted proposal to be a no-op, so that network retries and duplicate requests do not produce duplicate knowledge graph entries.

#### Acceptance Criteria

1. WHEN an authorized reviewer accepts a proposal that already has Status Accepted, THE Review_Service SHALL return a success response containing the proposal's existing ReviewedAt and ReviewedByUserId without creating additional Artifact, ArtifactFact, ArtifactRelationship, or SourceReference records and without modifying the proposal's ReviewedAt or ReviewedByUserId
2. WHEN an authorized reviewer rejects a proposal that already has Status Rejected, THE Review_Service SHALL return a success response containing the proposal's existing ReviewedAt and ReviewedByUserId without further state changes
3. IF an authorized reviewer attempts to accept a proposal that has Status Rejected, or reject a proposal that has Status Accepted, THEN THE Review_Service SHALL return an error indicating that a proposal in a conflicting terminal state cannot be transitioned to a different terminal state
4. WHEN processing an idempotent request for a proposal in a terminal state, THE Review_Service SHALL enforce the same authorization and visibility rules as for a Pending proposal — if the requesting user is not authorized or cannot see the proposal, THE Review_Service SHALL return a not-found or forbidden error as appropriate

### Requirement 11: Atomic Operations

**User Story:** As a system operator, I want proposal acceptance to be transactional, so that partial failures do not leave the knowledge graph in an inconsistent state.

#### Acceptance Criteria

1. WHEN a proposal acceptance involves creating or updating a knowledge graph entity plus creating a SourceReference plus updating the proposal status, THE Review_Service SHALL execute all operations within a single database transaction so that either all changes commit together or none are persisted
2. IF the database transaction fails during proposal acceptance, THEN THE Review_Service SHALL roll back all changes, leave the proposal in its pre-acceptance Status (Pending or Edited), and return an error response that includes the proposal Id and a reason indicating the operation could not be completed
3. IF a batch operation encounters a transaction failure for one proposal, THEN THE Review_Service SHALL process each proposal within its own independent transaction, report the failed proposal Id and error reason in the batch response's failed-proposals list, and continue processing remaining proposals in the batch without rolling back previously committed proposals
4. WHEN a proposal acceptance triggers a ReviewBatch Status transition (Pending to InReview, or InReview to Completed), THE Review_Service SHALL execute the batch status update outside the individual proposal's transaction so that a batch status update failure does not roll back the successfully accepted proposal

### Requirement 12: Campaign-Scoped Review Authorization

**User Story:** As a system operator, I want all review endpoints to require campaign membership, so that only authorized campaign participants can interact with review proposals.

#### Acceptance Criteria

1. THE API SHALL require campaign membership for all review endpoints by applying the CampaignMemberActionFilter, which resolves the CampaignMember from the route `campaignId` parameter and the authenticated user before any controller logic executes
2. IF the requesting user is not a member of the target campaign, THEN THE API SHALL return HTTP 403 Forbidden with an error response indicating access was denied
3. THE API SHALL derive the acting user identity from the authenticated JWT `sub` claim via the UserProvisioningMiddleware and SHALL NOT accept client-provided user identifiers for review operations
4. WHEN a review endpoint references a campaign that does not exist, THE API SHALL return HTTP 403 Forbidden (indistinguishable from the non-member response, so that campaign existence is not leaked to unauthorized users)
5. IF the `campaignId` route parameter is not a valid GUID, THEN THE API SHALL return HTTP 404 Not Found
6. IF the request does not contain a valid authenticated JWT, THEN THE API SHALL return HTTP 401 Unauthorized before the campaign membership check executes
